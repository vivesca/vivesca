# Agentic Engineering Patterns: Tool Design, Action Spaces, and the Evolution of AI Workflows Through 2025

This comprehensive report examines the emergence of agentic engineering as a distinct discipline for building software using coding agents, with particular emphasis on the design of tool systems, action spaces, and the architectural patterns that have proven effective in practice through 2024 and 2025. The analysis draws on the pioneering work of Simon Willison and the broader community exploring optimal approaches to agent-driven development, synthesizing practical patterns that have emerged from real-world implementations while addressing the fundamental challenges of managing AI agents as sophisticated software components within larger systems. The report reveals a fundamental shift in how developers conceptualize code generation and system development, moving from traditional prompt-based interaction toward carefully engineered loops where agents can reason, act, observe consequences, and iterate independently, fundamentally changing what is possible in software development.

## Understanding Agentic Engineering and Its Foundational Concepts

Agentic engineering represents a revolutionary approach to software development that fundamentally differs from both traditional programming and casual "vibe coding" with language models.[1][1] The term specifically refers to the practice of building software using coding agents—tools like Claude Code and OpenAI Codex—where the defining characteristic is that these agents can both generate and execute code independently, allowing them to test their output, iterate on failures, and solve problems without requiring constant human guidance for each step.[1][1] This capability to execute code and observe the results creates a feedback loop that fundamentally transforms the development experience from a linear conversation into a dynamic problem-solving process where the AI can recover from its own mistakes and refine its approach based on actual runtime feedback rather than theoretical analysis.

Simon Willison, a prominent figure in the AI-assisted programming space, has articulated the distinction between agentic engineering and what he terms "vibe coding." In his foundational work on Agentic Engineering Patterns, Willison emphasizes that agentic engineering sits at the opposite end of the spectrum from vibe coding.[1][1] Whereas vibe coding represents developers who pay minimal attention to the code being generated—often non-programmers using LLMs to produce working solutions—agentic engineering involves professional software engineers leveraging coding agents to amplify their existing expertise and accelerate their work.[1][1] This distinction is crucial because it establishes agentic engineering not as a replacement for software engineering skill but rather as an enhancement that requires deep domain knowledge to apply effectively. The professional engineer using agentic engineering tools needs to understand what is possible, what is worth building, how to design effective systems, and most critically, how to verify that the code produced actually works as intended.

The cost structure of software development has undergone a dramatic transformation through agentic engineering.[1][1][1] Willison identifies "writing code is cheap now" as the central challenge of the discipline because the marginal cost to produce initial working code has dropped to nearly nothing.[1][1][1] This represents not just a quantitative change but a qualitative shift in how developers must think about their work. When code was expensive to produce, teams prioritized careful planning, extensive code review, and minimal iteration. When code becomes cheap, the bottleneck shifts to validation, integration, and maintaining quality over time. This fundamental economic change reshapes the entire development philosophy—from one where you carefully craft each line knowing it will be scrutinized, to one where you rapidly generate multiple approaches, test them exhaustively, and select the proven winner.

## The Tool Use Design Pattern and Action Space Architecture

The tool use design pattern forms the backbone of effective agent systems, enabling AI agents to move beyond simple text generation to interact with external systems, databases, web services, and computational environments.[6][7] Understanding how tools are defined, discovered, and invoked represents essential knowledge for building practical agents. The tool use pattern operates through a well-defined flow where tools are described in a schema sent to the language model, the model selects appropriate tools and constructs calls with parameters, those calls are executed by the application, and results are fed back to the model, which incorporates this information into its response.[6][28]

Function calling, also known as tool calling, has become the primary mechanism through which modern agents interact with their environment.[6][28][32] Microsoft's formalization of the tool use design pattern identifies several essential building blocks that must work together seamlessly.[6] The first element is function or tool schemas, which are detailed definitions of available tools including function name, purpose, required parameters, and expected outputs.[6] These schemas serve a dual purpose: they enable the language model to understand what tools are available and what parameters are valid, while simultaneously enabling the application to validate that the model's requested invocations are well-formed before execution. The second element is function execution logic that governs how and when tools are invoked based on user intent and conversation context, which may include planner modules, routing mechanisms, or conditional flows that determine tool usage dynamically.[6] The third essential component is a message handling system that manages the conversational flow between user inputs, language model responses, tool calls, and tool outputs, maintaining state across multiple turns of interaction.[6]

Developers face crucial decisions about tool design philosophy when building agent systems. Simon Willison documents his practical approach to tool selection, noting that rather than relying heavily on the Model Context Protocol (MCP) for tool communication, he has found simple command-line tools and shell commands to be more effective for coding agents.[18][44] This observation reflects a broader principle: coding agents are remarkably effective at understanding how to use shell commands and existing CLI tools without requiring extensive token-heavy explanations.[17] When Willison needs to make a tool available to a coding agent, he often creates or references an AGENTS.md file that contains simple examples of how to use the relevant tools.[17] This lightweight approach proves more effective than complex tool schemas because language models can infer usage patterns from examples more efficiently than from formal specifications, and the agents can recover from mistakes through the feedback loop of trying commands, observing errors, and adjusting their approach.

The concept of an action space—the set of possible actions an agent can take—requires careful curation to balance capability with safety and coherence.[7][33] In the "arsenal" model described by developers working with frameworks like PocketFlow, the action space is conceived as a toolkit that agents select from based on the current situation.[7] The design of this action space involves three distinct tiers that scale in complexity and flexibility.[7] The first tier consists of simple binary commands or state-change signals, such as "request_human_help" or "finish_task," which are useful for clear, decisive actions and controlling overall workflow flow.[7] The second tier comprises parameterized tools that require specific inputs to function effectively, representing the most common and versatile actions in an agent's arsenal.[7] These are the sharpshooter's bow that needs proper aiming—the agent must not only choose the action but also provide correct parameters through carefully structured output, typically in formats like YAML or JSON.[7] The third tier represents programmable actions where the agent generates code or complex instructions on the fly, providing ultimate flexibility by allowing agents to solve novel problems they were never explicitly trained for.[7]

The quality of an agent's decision-making depends almost entirely on how clearly its action space is described.[7] A well-defined action space with sharp, unambiguous descriptions leads to intelligent, effective agent behavior, while a vague or confusing action space leads to errors and wasted iterations.[7] This principle extends beyond simple listing of available actions to include clear communication about when each action should or should not be used, what parameters are required, and what outcomes to expect. Documentation becomes a first-class engineering concern because it directly impacts agent performance.

## Designing Effective Agentic Loops

The concept of designing agentic loops represents one of the most consequential skills for practitioners working with coding agents.[17] Willison articulates his preferred definition of an AI agent as something that runs tools in a loop to achieve a goal, and the art of using agents well involves carefully designing both the tools available and the loop itself.[17] This perspective transforms agent building from a prompt engineering problem into a systems design problem where the architecture of the environment and tools matters as much as the quality of the underlying model.

Agentic loops operate most effectively when dealing with problems that have clear success criteria where finding a solution likely involves trial and error.[17] The pattern works particularly well for optimization tasks, refactoring scenarios, exploratory prototyping, and investigation work where multiple approaches might work and the best one needs to be discovered through experimentation rather than predetermined.[17] When developers find themselves thinking "ugh, I'm going to have to try a lot of variations here," this signals an excellent opportunity for an agentic loop.[17] Examples include optimizing Docker container sizes by trying different base images while keeping tests passing, investigating network bottlenecks by running load tests against different configurations, or exploring whether specific technologies fit project requirements through rapid prototyping.[17]

The design of an effective agentic loop requires several critical components working together harmoniously.[17] First, the environment in which the agent operates must be safe enough to allow autonomous exploration without risking real systems.[17] This often means creating dedicated sandboxes, test environments, or isolated resources specifically for the agent to experiment within.[17] Second, the tools available to the agent must be well-chosen to allow meaningful exploration while preventing futile paths.[17] Rather than providing every possible tool, successful loop design involves curating a focused set of tools that agents can effectively use. Third, credentials and permissions must be carefully scoped to match the task at hand—if an agent needs to call an API that costs money, the credentials should include budget limits that constrain spending even if the agent behaves erratically.[17]

Willison introduces "YOLO mode"—automatic approval of all commands—as a crucial enabler of effective agent performance despite its obvious risks.[17][22] When agents must request approval for every action, they become dramatically less effective because the interaction becomes tedious and the feedback loop slows to a crawl.[17] However, running in YOLO mode requires that the agent's environment be constrained so that even arbitrary code execution cannot cause unacceptable damage.[17] This represents a fundamental tradeoff in agent system design between safety and capability. Systems that are too constrained with excessive approval gates become useless because agents cannot explore effectively. Systems that grant too much autonomy without proper environmental constraints become dangerous. The sweet spot involves designing environments that allow unrestricted agent action within a clearly bounded scope where any damage is contained and acceptable.[17]

## Test-Driven Development and Red/Green Patterns for Agents

Test-first development has emerged as one of the most effective patterns for improving code quality when working with coding agents.[1][16][20] The red/green TDD (Test Driven Development) pattern involves writing automated tests first, confirming they fail (the red phase), then implementing code to make tests pass (the green phase).[16] This approach proves to be a fantastic fit for coding agents because it provides built-in verification that the generated code actually works and creates a clear specification that the agent can work toward.[1][16][20]

The advantages of test-first development with agents manifest in multiple ways.[20] Automated tests are no longer optional when working with coding agents because they provide the only reliable way to verify that AI-generated code functions correctly.[20] The old excuses for not writing tests—that they are time-consuming and expensive when a codebase is rapidly evolving—no longer hold when an agent can generate a complete test suite in minutes.[20] Furthermore, agents themselves are already biased toward testing; the presence of an existing test suite nearly guarantees that agents will run tests after making changes and will expand the test suite with new tests for code they generate.[20] This represents a virtuous cycle where good testing practices are reinforced rather than fought against.

Simple four-word prompts encapsulate substantial software engineering discipline that is already baked into frontier models.[1][16][20] When Willison prompts a coding agent with "Use red/green TDD," the model understands this refers to a comprehensive testing methodology involving test-first development, confirming tests fail before implementation, and ensuring the final tests pass.[16] Similarly, "First run the tests" serves multiple purposes: it signals that a test suite exists, forces the agent to learn how to run it, places the agent in a testing mindset, and creates an almost certain commitment to running tests in the future to verify changes.[20] These concise prompts work because the frontier models have encountered extensive examples of TDD in their training data and understand the pattern deeply.

The combination of code cheapness and powerful testing enables what Willison calls zero tolerance for code quality issues.[4][4] When refactoring tasks that are conceptually simple but time-consuming—such as fixing API designs that impact dozens of files, updating nomenclature throughout a codebase, or splitting oversized modules into smaller components—can be delegated to a coding agent with test coverage ensuring correctness, developers can finally justify doing this maintenance work as part of normal development rather than relegating it to distant future cycles.[4][4] The compound effect of continuously improving code quality while shipping new features represents a fundamental shift in software development practices enabled by agentic engineering.

## Multi-Agent Architectures and Coordination Patterns

As agentic systems have matured through 2024 and 2025, sophisticated multi-agent architectures have emerged to handle increasingly complex tasks that benefit from specialized roles and parallel execution.[8][9][23] The five most popular agentic AI design patterns identified for enterprise use include Task-Oriented Agents focused on single high-value tasks, Multi-Agent Collaboration where specialized agents work together, Self-Improving Agents that adapt through feedback loops, RAG Agents that ground responses in enterprise knowledge, and Orchestrator Agents that manage workflow and coordination.[8]

Multi-agent collaboration represents a paradigm shift from the single-agent problem-solving approach by conceptualizing AI systems as teams of specialists rather than monolithic intelligences.[8][9][23] In a demand forecasting and price optimization scenario, the architecture might include an Ingestion Agent pulling data from ERP and POS systems, a Forecasting Agent predicting demand for each product, an Optimization Agent suggesting prices based on demand and margins, a Policy Agent checking compliance with rules and regulations, and an Explainer Agent generating clear summaries for decision-makers.[8] Each agent maintains its own focused scope, operates with specialized knowledge, and contributes a specific piece of the solution. This modular approach provides several advantages: clearer responsibility boundaries prevent duplication and confusion, parallel execution of independent agents accelerates overall solution time, and individual agent failures can be isolated rather than cascading through the entire system.[8][9]

Research from Anthropic demonstrates the effectiveness of multi-agent architectures at scale, showing that a multi-agent system with Claude Opus 4 as the lead agent and Claude Sonnet 4 subagents outperformed single-agent Claude Opus 4 by 90.2 percent on internal research evaluations.[23] This performance improvement stems from the architecture's ability to distribute work across agents with separate context windows, enabling parallel reasoning that a single agent operating within a context window cannot achieve.[23] The separate context isolation prevents context bloat where accumulated information from previous steps drowns out the signal for current decisions.

LangChain's analysis of multi-agent architectures identifies four primary patterns that form the foundation of most modern multi-agent applications.[23] The subagents pattern uses a supervisor agent coordinating specialized subagents by calling them as tools, maintaining conversation context in the main agent while subagents remain stateless.[23] The skills pattern enables an agent to dynamically adopt specialized personas by loading relevant prompts and knowledge on demand, providing progressive disclosure of capabilities.[23] The router pattern classifies incoming requests and dispatches them to specialized agents executing queries in parallel before synthesizing results.[23] The handoff pattern manages sequential workflow with state transitions where agents converse with users throughout the process and coordinate handoffs between specialized agents.[23] Each pattern involves different tradeoffs regarding token efficiency, latency, parallelization capability, and state management.

Context management across multi-agent systems represents a critical engineering challenge that has received substantial attention through 2025.[26][44] Google's Agent Development Kit (ADK) introduces a novel architecture treating context as a compiled view over richer stateful systems rather than a mutable string buffer.[26] In this model, sessions, memory, and artifacts serve as sources maintaining the full structured state, while flows and processors implement a compiler pipeline transforming that state, and the working context represents the compiled view shipped to the language model for each invocation.[26] This separation enables optimization through context caching where stable prefixes like system instructions and long-lived summaries can be cached separately from variable suffixes with new tool outputs, dramatically reducing token consumption for long-running agentic systems.[26]

Careful scope management prevents agents from being overwhelmed by irrelevant information.[26][44] Rather than flooding each model call with complete history, effective systems implement tiered context architecture where human domain knowledge defines the structure—where data lives, how it is summarized, and what filters apply—while agents dynamically decide when to reach for specific memory blocks or artifacts.[26] This negotiation between human architecture and agent intelligence produces optimal working contexts that are neither too sparse to accomplish tasks nor so dense that signal-to-noise ratios degrade performance.

## Prompting Strategies and Context Engineering

The evolution of prompting techniques represents a substantial refinement in how developers communicate with agents, moving from ad-hoc prompts toward systematic context engineering disciplines.[19][30][44] Modern prompt engineering recognizes that the entire context supplied to a model—including system instructions, tool definitions, tool outputs, user instructions, and model outputs from previous turns—all contribute to agent performance.[19] Anthropic's guidance on effective context engineering emphasizes that system prompts should be extremely clear and use simple, direct language presenting ideas at the right altitude for the agent.[30] This "right altitude" represents the Goldilocks zone between two common failure modes: at one extreme, engineers hardcode complex brittle logic into prompts trying to elicit exact behavior, while at the other extreme, prompts provide insufficient information for agents to understand expectations.

The best practice approach involves starting with minimal prompts using the best available model and adding instructions iteratively based on failure modes observed during testing.[30] Prompts should be organized into distinct sections like background_information, instructions, tool_guidance, and output_description, using XML tagging or Markdown headers to delineate sections.[30] While exact formatting decreases in importance as models improve, consistency remains critical because inconsistencies confuse models and degrade performance. Developers should strive for the minimal set of information that fully outlines expected behavior—minimal does not mean short, but rather avoiding unnecessary elaboration while ensuring sufficient information exists for the agent to adhere to desired behavior.[30]

Context engineering extends beyond static system prompts to include dynamic context management throughout agent execution.[44] LangChain identifies four primary strategies for context engineering: writing context through saving information outside the context window to help agents perform tasks, selecting context through pulling relevant information into the window when needed, compressing context to retain only essential tokens for current tasks, and isolating context through splitting it across subagents.[44] Scratchpads provide one effective mechanism where agents can persist notes and remembered information for future related tasks, allowing saving of information outside the context window while keeping it available for reference.[44] This pattern mimics human problem-solving where we take notes as we work, knowing we can reference them later, preventing the need to repeatedly store all information in active memory.

The tension between providing comprehensive context and staying within token limits shapes modern agent architecture.[44] Longer context enables agents to reason more effectively with more information, but token costs increase linearly while latency increases, and paradoxically, very long contexts can actually degrade agent performance as signal-to-noise ratios deteriorate.[44] Successful systems implement summarization strategies where after extended interactions, agent trajectories are compressed through recursive or hierarchical summarization, preserving essential information while reducing token consumption.[44] Claude Code implements this through "auto-compact" which automatically triggers context summarization after exceeding 95 percent of the context window, demonstrating that this pattern is now central to mainstream agent tooling.

## Reasoning, Reflection, and Advanced Reasoning Patterns

A major breakthrough in agent capabilities emerged through 2025 with the introduction of reasoning and thinking modes to frontier models.[15][15][42] Reasoning enables models to spend additional time generating text that talks through problems and potential solutions before presenting final responses, functioning similar to a person thinking out loud while working through complex problems.[15][15] This capability particularly benefits debugging scenarios where models can navigate complex code paths, mixing tool calls with reasoning to follow function calls back to potential issue sources.[15] Many coding agents now include options to dial up or down reasoning effort level, encouraging models to spend more time on harder problems through techniques like chain-of-thought prompting and tree-of-thought exploration.[15]

The ReAct (Reasoning and Acting) framework combines reasoning and tool calling in an interleaved manner, allowing models to generate both reasoning traces and task-specific actions.[35][37] ReAct prompts models to generate verbal reasoning about what to do, select an action from available tools, observe the results, and iterate based on observations.[35][37] This pattern has shown particular effectiveness for tasks requiring exploration, strategic lookahead, or where initial decisions significantly impact final outcomes.[37] The structured format of thought-action-observation cycles enables models to induce and update action plans, handle exceptions, and incorporate external information from tools into reasoning.[35][37]

More advanced reasoning patterns continue to emerge from research communities studying LLM agents. Tree of Thoughts (ToT) generalizes chain-of-thought prompting by enabling exploration over coherent units of text that serve as intermediate reasoning steps, allowing models to consider multiple reasoning paths and self-evaluate choices to decide next course of action, with ability to backtrack when necessary.[36] Where chain-of-thought only considers single left-to-right token sequences, tree search enables deliberate decision making considering multiple paths, achieving dramatic improvements on reasoning-intensive tasks like Game of 24 where GPT-4 with standard prompting solved only 4 percent but ToT achieved 74 percent success rate.[36]

## Security, Sandboxing, and Agent Governance

As agents have been deployed into production systems handling increasingly sensitive tasks, security and sandboxing have become critical engineering concerns. The "lethal trifecta" for agent security comprises three dangerous capability combinations: access to private data, exposure to untrusted content, and the ability to communicate externally. When agents possess all three capabilities, prompt injection attacks become catastrophic because attackers can embed malicious instructions in untrusted content, causing agents to exfiltrate private data through external communications. This represents a fundamental security challenge: we still do not know how to 100 percent reliably prevent prompt injection attacks, meaning guardrails claiming 95 percent protection remain inadequate since 95 percent is a failing grade in security contexts.

The practical approach to agent security involves accepting that perfect solutions do not exist and instead carefully constraining agent capabilities to acceptable tradeoff points. Sandboxing implementations must isolate code execution in secure environments preventing unauthorized access and system compromise, but this is substantially more complex than traditional containerization because agents generate and execute arbitrary code. Standard containers share the host kernel with running code, exposing it to any kernel vulnerabilities exploitable by code executed within the sandbox. Production-grade agent sandboxing requires isolation layers that completely separate agent execution from host systems through approaches like Firecracker microVMs providing hardware-level isolation with dedicated kernels per workload. Firecracker creates lightweight virtual machines booting in approximately 125 milliseconds with less than 5 megabytes overhead per VM, enabling efficient multi-tenant agent execution where each agent runs with its own isolated kernel and cannot escape to compromise the host.

Alternative approaches include gVisor which mediates system calls via a separate user-space kernel, providing syscall-level isolation without the overhead of full virtual machines, and Kata Containers which combine microVM properties with standard container workflows. Each approach represents different points on the tradeoff spectrum between isolation strength and performance overhead. For production systems running untrusted AI-generated code, Firecracker microVMs or Kata Containers provide the strongest isolation, while gVisor is acceptable for compute workloads where code is controlled, and standard containers should only be used for trusted code.

Beyond sandboxing, governance frameworks must address permission scoping, approval workflows, and observability. Agents should only have access to credentials and permissions required for their specific task, implemented through secret injection approaches preventing credentials from being shared with agents through environment variables. Network access should operate under a zero-trust model where all connections are explicitly allowed rather than implicitly permitted, with egress filtering blocking all outbound connections by default and whitelisting only required API endpoints. Every instance of specific high-risk actions should require fresh user approval—importantly, approvals should never be cached or persisted because a single cached approval opens doors to future adversarial abuse. Comprehensive observability through unified telemetry frameworks linking application logs, infrastructure metrics, and AI model decisions into traceable transaction IDs remains essential because debugging agentic systems demands understanding both what code executed and what reasoning led to that code execution.[49]

## Patterns for Tool Integration and Execution

Effective tool integration requires careful attention to how agents discover, understand, and invoke tools.[32] The Model Context Protocol (MCP) was developed to standardize agent-tool communication, enabling dynamic tool discovery at runtime rather than fixed tool definitions baked into applications.[32] With MCP, agents can query servers to discover available tools and choose appropriate ones dynamically based on user queries, decoupling LLM applications from fixed tool sets and enabling modularity and extensibility especially valuable for complex or evolving agent systems.[32] However, as Willison documents through 2025, many practitioners have found that simple CLI tools and shell commands remain more effective for coding agents than complex MCP implementations.[18][44] This observation reflects that language models can effectively reason about command-line tool usage through examples better than parsing formal tool schemas, and the agent can recover from mistakes through feedback loops of execution and observation.

Tool documentation limitations represent a significant source of agent hallucinations where agents develop incorrect mental models of tool behavior based on misleading or incomplete documentation. Agents often misunderstand what tools do, resulting in confident invocation of invalid tools or incorrect parameter usage. When tool documentation lacks clarity about edge cases, optional parameters, or limitations, agents struggle to construct correct invocations. This connects directly to the importance of well-written tool schemas and clear error messages—when tools fail, the error messages should help agents understand why and how to correct their approach rather than leaving them to guess.

Pure function invocation represents an emerging best practice where infrastructure-oriented tasks shift to pure functions rather than tool calls, reserving tool calls for operations where language-driven reasoning is genuinely necessary.[46] This pattern improves workflow predictability because pure functions provide deterministic behavior, reducing token consumption by avoiding the overhead of prompting models to make decisions about infrastructure tasks that should follow algorithmic logic.[46] A one-agent-one-tool design whenever tool usage is required creates predictable roles and simplifies prompting by eliminating tool-selection noise, allowing agents to focus on parameter inference and execution rather than choosing between multiple similar tools.[46]

## Real-World Applications and Case Studies

The progression of agentic engineering from experimental technique to production deployment through 2024 and 2025 is illuminated through several compelling case studies demonstrating patterns that work at scale. StrongDM's AI team, starting with a radical principle of "no hand-coded software," encountered and solved the fundamental validation problem of how to prove that software produced entirely by coding agents actually works. Their answer, inspired by scenario testing, involved repurposing the word "scenario" to represent end-to-end user stories that could be understood and flexibly validated by language models, stored outside the codebase similar to holdout sets in model training to prevent agents from cheating by overfitting to test cases. This approach transitions validation from boolean definitions of success—test suite is green—to probabilistic and empirical ones, quantifying "satisfaction" as the fraction of observed trajectories through scenarios that likely satisfy users.

StrongDM's Digital Twin Universe concept extends this validation approach by creating independent clones of services where testing agents could execute scenarios without rate limits or quotas. Rather than testing against production systems, agents test against simulated environments with identical behavior, enabling exhaustive testing and rapid iteration. This represents a sophisticated evolution of agent validation that acknowledges hallucination and unreliability remain challenges even with state-of-the-art models, but channels those challenges into environments where hallucinations cannot cause real damage while still providing comprehensive proof of correctness.

Orbital Witness's SRE team documented their evolution in working with coding agents, articulating that the fundamental role shift involves moving from writing code to validating code—AI generates, humans prove it works. Their two-part proof system requires both manual testing where developers personally run code against real data and verify outputs, and automated testing with comprehensive coverage, with the critical insight that manual proof establishes initial validation and automated tests reinforce it rather than replacing manual testing. This pattern reflects broader industry recognition that even excellent coding agents sometimes produce code that passes tests but fails in real deployment, necessitating manual verification that code actually achieves its intended purpose before being considered complete.

Harper Reed's LLM codegen workflow documents end-to-end processes from brainstorming through specification generation to breaking specifications into appropriately-sized implementation steps, generating prompts for code-generation LLMs that implement each step in test-driven manner, and executing prompts using tools like aider or Cursor.[50] The workflow emphasizes that success depends most on specification quality—well-specified work requires minimal prompting iteration while poorly-specified work leads to repeated clarifications and rework.[50] The iterative refinement of specifications and decomposition into right-sized steps that build on each other, with no hanging code left unintegrated, demonstrates that agentic engineering success requires upfront investment in clear specification and planning.

## Emerging Trends and Future Directions

Throughout 2025, several trends have solidified as core practices in agentic engineering workflows. Simon Willison's transition from skepticism about agents to embracing "the parallel coding agent lifestyle" captures the evolution in thinking as practitioners discovered that firing up multiple agents simultaneously—whether against the same repository or different checkouts—enables efficient parallelization of exploratory work without overwhelming human review capacity.[22] Research tasks that answer questions without modifying production code, proof-of-concept explorations of new libraries and technologies, and documentation and explanation generation can proceed in parallel while primary development continues, with humans reviewing outputs asynchronously.[22]

The conformance suite approach for new technologies and programming languages represents a crucial unlock for encoding domain-specific knowledge in ways agents can effectively use.[11] Testing against comprehensive conformance suites gives frontier models clear success criteria and abundant examples of correct behavior, making them significantly more effective than unstructured specifications alone.[11] This observation suggests that any organization releasing new protocols, programming languages, or major frameworks should consider including language-agnostic conformance test suites as standard practice, not as optional testing infrastructure but as essential scaffolding for agent adoption.

The development of Claude Skills as an alternative to MCP demonstrates ongoing refinement of how specialized knowledge is encoded for agents.[18] Skills represent Markdown files in folders optionally accompanied by executable scripts, providing extremely simple mechanisms for making new abilities available to models without complex protocol overhead.[18] The radical simplicity proves powerful—because skills depend only on filesystem access and command execution, not specialized protocol servers, they work across different coding agent platforms and models simply by pointing agents at skill folders.[18] This simplicity has led some to question whether skills represent genuine features or merely formalize existing practices of dropping instructions into files for agents to read, but the practical effectiveness suggests that sometimes formalizing and naming patterns matters more than their technical novelty.

Agentic RAG systems combining retrieval-augmented generation with agent reasoning represent a major direction for enterprise adoption, enabling agents to ground responses in organization-specific verified data sources while reducing hallucination risk.[8][45] Rather than relying entirely on model training data, RAG agents retrieve relevant information from enterprise knowledge sources at runtime, ensuring responses reflect current facts rather than potentially stale training data.[8] This pattern particularly benefits scenarios where accuracy to real-time information matters more than generalization—customer support, financial analysis, research assistance.

The concept of compound engineering, documented by Every's team working with coding agents, emphasizes that agent interactions should conclude with retrospectives that document what worked so it can be reproduced in future runs.[4][4] Small improvements compound over time as each project builds on lessons learned from previous ones, with quality enhancements that were previously too time-consuming to justify becoming standard practice when agents can execute them in minutes.[4][4]

## Conclusion: Synthesizing the Patterns and Looking Forward

The emergence of agentic engineering as a distinct discipline represents a fundamental reconfiguration of software development practice that has been taking shape through 2024 and 2025. Simon Willison's contributions in documenting patterns, establishing best practices, and articulating principles like "code is cheap now" and "first run the tests" have proven remarkably prescient and aligned with discoveries across the broader community. The recognition that professional software engineers amplifying their expertise through coding agents represents a distinct practice from casual vibe coding establishes appropriate expectations about required skill levels and the role that human judgment plays in agent-driven development.

The sophisticated evolution of tool design from fixed schemas to flexible CLI tool composition, the emergence of multi-agent architectures that coordinate specialized agents more effectively than monolithic approaches, and the maturation of security frameworks addressing the lethal trifecta of combined capabilities all demonstrate that agentic engineering is graduating from experimental technique to engineering discipline. The development of validation patterns through scenario testing, holdout sets, and Digital Twin Universes indicates that the field has recognized and begun addressing the central challenge: proving that agent-generated code works without requiring exhaustive human review of every line.

The patterns documented here—test-first development, designing effective agentic loops, careful context engineering, appropriate tool curation, and multi-agent coordination—have been validated through production deployments across numerous organizations and appear to represent genuine solutions rather than temporary expedients. As models continue to improve and agentic engineering tools proliferate, the bottleneck will shift increasingly from raw code generation capability toward validation, integration, and ensuring produced systems align with business intent. Organizations that master the practices documented here—specifying clearly, validating thoroughly, maintaining comprehensive test suites, and orchestrating multiple specialized agents effectively—will find themselves able to deliver software faster and with higher quality than traditional approaches allow.

The future of agentic engineering likely involves further specialization of agents into roles with clear responsibilities, increasing sophistication in how context flows through multi-agent systems, more formalized approaches to encoding domain knowledge through conformance suites and skills, and continued maturation of security frameworks enabling agents to operate in production environments safely. The fundamental question that will guide this evolution is one articulated by StrongDM: how can we prove that software we are producing works if both implementation and tests are written by autonomous agents? The answers emerging from leading practitioners suggest that comprehensive validation, scenario-based testing against specifications, and proof-first approaches focused on demonstrated functionality rather than code inspection will continue defining the discipline. As this practice matures, agentic engineering will likely become not a specialized skill reserved for early adopters but rather the default approach for professional software development teams seeking to move faster while maintaining and improving quality over time.
