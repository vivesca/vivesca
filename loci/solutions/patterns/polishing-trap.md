# The Polishing Trap

Internal documents with known audiences don't need perfection. Set a ship threshold before starting, not after the 5th editing pass.

## The Pattern

1. First pass gets to ~80% quality in ~20% of total time
2. Each subsequent pass finds one more thing to tweak (there's always one)
3. Councils/reviews amplify: they're designed to find issues, so they always will
4. No "done" criteria → infinite loop of diminishing returns
5. Anxiety from one genuine fix (e.g. governance overclaim) triggers hyper-vigilance on everything

## Case Study: Capco CV (Feb 2026)

- 2-slide internal CV for partner staffing
- ~12 hours across 7 days, ~15 council runs (~$8-10)
- Genuine fixes (governance accuracy, de-editorialising) took ~2 hours
- Remaining ~10 hours: wordsmithing, bracket removal, arrow symbols, council-validating incremental changes
- Audience (Capco partners) skims in 30 seconds

## The Fix

Before starting any document work, set the ship threshold:

1. **Who reads this?** Internal = lower bar. Client-facing = higher bar.
2. **What decision does it drive?** Staffing = facts matter, prose doesn't. Pitch = both matter.
3. **Define "done" in one sentence.** E.g. "Facts accurate, no overclaims, readable in 30 seconds."
4. **One council as quality gate, not a comfort blanket.** If the first council says ship, ship.
5. **Time-box.** Internal docs: 2-3 hours max. Client-facing: up to a day.

## Signals You're in the Trap

- Running a council after every change
- Debating word choices that don't change meaning ("suppressed" vs "hibernated" vs "identified for hibernation")
- Multiple sessions on the same document across multiple days
- Building infrastructure (skills, reusable prompts) for a one-time deliverable before shipping it

## Claude's Role

Push back when polishing has entered diminishing returns. Say: "This is past the ship threshold — send it." Don't enable the loop by suggesting one more tweak.
